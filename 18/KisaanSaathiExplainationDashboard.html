📋 Summary
Kisaan Saathi is a rule-based agricultural advisory platform that combines live NPK/pH sensor readings, ICAR/STCR fertilizer guidelines, short-term weather forecasts, and farmer-specific application history to generate personalised fertilizer and input recommendations. A structured Claude API layer converts these computed recommendations into validated JSON and farmer-friendly WhatsApp/Hindi output, while village agents and stored records enable continuous field use and long-term soil monitoring.
⚙️ System pipeline
Step 1
📥 Input capture
Farmer ID, crop, crop stage, land size, village/location, NPK/pH sensor values, previous fertilizer/input applications.
Step 2
🌦️ Weather fetch
Rainfall probability, temperature, and humidity for the next few days.
Step 3
📐 Rule engine
Apply ICAR/STCR-based nutrient logic and crop-stage rules — fertilizer type, quantity, timing, and method.
Step 4
🕐 History adjustment
Check what the farmer already applied; avoid duplicate or excessive recommendations.
Step 5
🛡️ Risk filter
Modify or delay advice if rain or other conditions make application inefficient or unsafe.
Step 6
🤖 Claude layer
Returns final advisory in a fixed JSON schema, not free text. Converts to Hindi/WhatsApp-ready blocks.
🧩 Core modules
📡
Sensor module
ESP32 + NPK + pH probe capture and optional phone/Bluetooth sync.
👤
Farmer profile module

🌾
Advisory engine
Deterministic ICAR-guided rule system, not ML.
🌤️
Weather module
Fetches forecast and creates simple risk flags, e.g. "avoid urea before heavy rain."
🤖
Claude formatting layer
Receives computed facts and returns structured JSON in a schema-safe format.
💬
WhatsApp interface
Farmer or agent submits values and receives actionable guidance.
📊
Admin / dashboard layer
Stores readings, farmer records, and village-level soil map over time.
🤖 Claude's role — four controlled jobs
1 Validate or fill a structured response schema — Claude can be constrained to JSON schema outputs.
2 Convert computed recommendations into human-readable explanations.
3 Produce Hindi/English advisory text from the already-computed rule-engine result.
4 Generate consistent WhatsApp-ready response blocks: summary, dosage, caution, next step, and follow-up date.

⚖️ Decision logic — history-aware & weather-aware checks
🔁 Same fertilizer already applied recently — avoid duplicate recommendation.
⏱️ Maximum safe interval not completed — suppress or delay.
⚠️ Repeated nitrogen recommendation within too short a window — blocked.
🚫 Incompatible combination risk — flagged before delivery.
🌧️ Weather-aware suppression: delay application before forecast heavy rain (e.g. urea wash-off risk).

⚠️ Risks & blockers
⚠️ ICAR/STCR rule encoding requires agronomic domain expertise — no ML fallback if rules are incomplete or a crop is not covered.
⚠️ Weather API reliability in rural or low-connectivity areas may degrade forecast accuracy.
⚠️ WhatsApp API access requires Meta Business approval and compliance — approval timeline is not specified.
⚠️ Sensor hardware (ESP32 + NPK + pH probe) in the field faces durability, calibration, and connectivity challenges.
⚠️ Hindi voice text quality and local dialect accuracy not validated yet.
⚠️ Village agent training and sustained engagement are critical — dropout risk during pilot.
🔒 Key design constraints
🔒 Agronomy logic must stay outside the LLM — Claude is a formatting and translation layer only, not a recommendation engine.
🔒 Rule-based first, LLM-assisted second — hallucination risk means no free-text fertilizer recommendations from Claude.
🔒 History-aware checks must prevent duplicate nitrogen/urea recommendations within unsafe intervals — enforced in the rule engine, not assumed.
🔒 System must be auditable — deterministic rules keep the advisory traceable and verifiable by agronomists.
